# Coding Conventions

**Analysis Date:** 2026-03-02

## Naming Patterns

**Files:**
- PascalCase for code-behind files: `App.xaml.cs`, `MainWindow.xaml.cs`, `AssemblyInfo.cs`
- XAML markup files use PascalCase: `App.xaml`, `MainWindow.xaml`
- Generated code files use suffix `.g.cs`: `App.g.cs`, `MainWindow.g.cs`
- Assembly info files: `AssemblyInfo.cs`

**Classes:**
- PascalCase for all classes: `App`, `MainWindow`, `IComponentConnector`
- Partial classes used for XAML code-behind separation: `public partial class MainWindow : Window`

**Methods:**
- PascalCase for public methods: `InitializeComponent()`, `Connect()`
- camelCase for private methods (follow C# standard)
- Constructor naming follows class name exactly: `public MainWindow()`

**Properties:**
- PascalCase for public properties
- camelCase for private fields: `_contentLoaded`

**Namespaces:**
- Project namespace: `JoJot` (single root namespace)
- Global usings enabled, so full namespace paths not always required in generated code

## Code Style

**Formatting:**
- .NET 10 framework conventions
- Implicit usings enabled (`using System;`, `using System.Collections.Generic;` etc. are auto-added)
- Nullable reference types enabled (`<Nullable>enable</Nullable>`)
- Standard C# indentation with braces on same line for methods

**Example Structure:**
```csharp
namespace JoJot
{
    /// <summary>
    /// Interaction logic for MainWindow.xaml
    /// </summary>
    public partial class MainWindow : Window
    {
        public MainWindow()
        {
            InitializeComponent();
        }
    }
}
```

**Class Modifiers:**
- `public partial` for XAML code-behind classes
- Full access modifiers specified explicitly
- Generated code includes attributes like `[DebuggerNonUserCodeAttribute()]` and `[GeneratedCodeAttribute()]`

## Import Organization

**Order:**
1. System namespace imports first: `using System;`, `using System.Text;`, `using System.Configuration;`
2. System.X subnamespace imports: `using System.Windows;`, `using System.Windows.Controls;`
3. Local namespace imports: `using JoJot;`

**Pattern Observed:**
```csharp
using System.Text;
using System.Windows;
using System.Windows.Controls;
using System.Windows.Data;
using System.Windows.Documents;
using System.Windows.Input;
using System.Windows.Media;
using System.Windows.Media.Imaging;
using System.Windows.Navigation;
using System.Windows.Shapes;

namespace JoJot
{
    // Class definition
}
```

**Global Usings:**
- System
- System.Collections.Generic
- System.Linq
- System.Threading
- System.Threading.Tasks

These are auto-generated in `JoJot.GlobalUsings.g.cs` and available throughout the project.

## Comments

**XML Documentation:**
- Use `/// <summary>` XML doc comments for public classes and methods
- Format: `/// <summary>\n/// [Description]\n/// </summary>`
- Example from codebase:
```csharp
/// <summary>
/// Interaction logic for MainWindow.xaml
/// </summary>
public partial class MainWindow : Window
```

**Inline Comments:**
- Used sparingly in generated code with attribute explanations
- Example: `// where theme specific resource dictionaries are located`

**Generated Code Comments:**
- Assembly-level comments explain purpose in `AssemblyInfo.cs`
- `<auto-generated/>` and `<auto-generated>` tags distinguish generated vs. manual code

## XAML Conventions

**Namespace Declarations:**
```xaml
xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
xmlns:x="http://schemas.microsoft.com/expression/blend/2008"
xmlns:local="clr-namespace:JoJot"
```

**Class Declaration:**
```xaml
<Window x:Class="JoJot.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/expression/blend/2008"
```

**Root Element Properties:**
- `x:Class` attribute links XAML to code-behind class
- `Title`, `Height`, `Width` properties specified in opening element
- `mc:Ignorable="d"` for design-time attributes

## Project Configuration

**Target Framework:** `net10.0-windows`

**Language Features Enabled:**
- Nullable reference types: `<Nullable>enable</Nullable>`
- Implicit usings: `<ImplicitUsings>enable</ImplicitUsings>`
- WPF framework: `<UseWPF>true</UseWPF>`
- Windows desktop application: `<OutputType>WinExe</OutputType>`

**Attributes in Generated Code:**
- `[DebuggerNonUserCodeAttribute()]` - Generated code marked for debugger skip
- `[GeneratedCodeAttribute()]` - Identifies auto-generated sections
- `[EditorBrowsableAttribute(EditorBrowsableState.Never)]` - Hides generated code from IntelliSense
- `[SuppressMessageAttribute()]` - Suppresses code analysis warnings for generated code

## Error Handling

**Strategy:** Not yet defined (minimal code in current project)

**Current Approach:**
- Standard .NET exception handling patterns (implicit)
- Generated code includes defensive checks: `if (_contentLoaded) { return; }`
- No explicit try-catch blocks in current source files

## Module Design

**Partial Classes:**
- Code-behind uses partial classes: `public partial class App`, `public partial class MainWindow`
- Allows separation of generated XAML initialization code from manual code-behind logic

**Visibility:**
- Public classes for UI components (windows, applications)
- Generated internal classes remain internal
- Private fields use underscore prefix: `_contentLoaded`
