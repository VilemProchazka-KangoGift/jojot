## Current Code State

### Fix 1: OnClosing (line 1650)
```csharp
protected override void OnClosing(CancelEventArgs e)
{
    base.OnClosing(e);
    _autosaveService.Stop();
    _checkpointTimer.Stop();
    // Synchronous content flush exists...
    // BUT: No CommitPendingDeletionAsync call
    // Geometry save happens after content flush
}
```
**Gap:** If user closes window (X button) while a deletion toast is visible, the pending deletion is never committed. Tabs reappear on next launch.

**Fix:** Add `CommitPendingDeletionAsync().GetAwaiter().GetResult()` before the geometry save block. The `.GetAwaiter().GetResult()` pattern is already used on line 1667-1668 for content flush -- consistent pattern.

### Fix 2: ShowDragOverlayAsync (line 3186)
```csharp
private async Task ShowDragOverlayAsync(string fromGuid, string toGuid, string toName)
{
    if (_isDragOverlayActive) return;
    _isDragOverlayActive = true;
    // Sets drag state fields...
    await DatabaseService.InsertPendingMoveAsync(_desktopGuid, fromGuid, toGuid);
    // No FlushAsync before the InsertPendingMoveAsync call
}
```
**Gap:** Up to 500ms of unsaved content exists when drag is detected. If app crashes during drag state, content since last autosave is lost.

**Fix:** Add `await _autosaveService.FlushAsync()` before the `InsertPendingMoveAsync` call. FlushAsync is already used in tab-switch path (TabList_SelectionChanged) -- proven pattern.

### Fix 3: FlushAndClose (line 1627)
```csharp
public void FlushAndClose()
{
    // Unsubscribe, stop timers...
    _ = CommitPendingDeletionAsync();  // Fire-and-forget!
    SaveCurrentTabContent();
    Close();
}
```
**Gap:** `_ = CommitPendingDeletionAsync()` is fire-and-forget. `ExitApplication` calls `FlushAndClose` on each window, then immediately calls `Environment.Exit(0)`. The deletion may not complete before process termination.

**Fix:** Change to synchronous await: `CommitPendingDeletionAsync().GetAwaiter().GetResult()`. This is safe because:
- `FlushAndClose` is called from `ExitApplication` (menu exit path) which runs on UI thread
- The method is `public void` (not async) -- callers expect synchronous completion
- `.GetAwaiter().GetResult()` is the established pattern in `OnClosing` (line 1667-1668)
- `CommitPendingDeletionAsync` is fast (just DB deletes) -- no UI freeze concern

## Architecture Patterns

### Synchronous Flush on Close Pattern
The codebase consistently uses `.GetAwaiter().GetResult()` when async operations must complete before window closure:
- `OnClosing` line 1667: `DatabaseService.UpdateNoteContentAsync(...).GetAwaiter().GetResult()`
- This is the correct WPF pattern for synchronous async completion in lifecycle events

### FlushAsync Guard Pattern
`FlushAsync` is used at tab-switch boundaries. It internally stops the timer and saves. Adding it before drag detection follows the same "flush before state transition" pattern.

## Common Pitfalls

### Pitfall 1: Deadlock with .GetAwaiter().GetResult()
**What goes wrong:** Calling `.GetAwaiter().GetResult()` on a task that captures the synchronization context can deadlock.
**Why it happens:** WPF has a single-threaded synchronization context. If the awaited task posts back to the UI thread, deadlock occurs.
**How to avoid:** `CommitPendingDeletionAsync` only does DB operations (no UI thread marshaling), and `DatabaseService.DeleteNoteAsync` uses `ConfigureAwait(false)` or direct SQLite calls. The existing `UpdateNoteContentAsync.GetAwaiter().GetResult()` on line 1667 proves this pattern is safe in this codebase.
**Warning signs:** UI freeze on window close.

### Pitfall 2: Double CommitPendingDeletionAsync
**What goes wrong:** Adding the call to both OnClosing and FlushAndClose could lead to double-commit attempts.
**Why it's safe:** `CommitPendingDeletionAsync` has a null guard (`if (_pendingDeletion == null) return;`) and captures-then-clears the field before any awaits. Second call is a no-op.
