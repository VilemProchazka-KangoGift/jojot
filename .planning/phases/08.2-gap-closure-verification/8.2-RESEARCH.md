## Verification Format (Established Pattern)

Existing VERIFICATION.md files (Phases 1-5) follow a consistent structure:

### Frontmatter
```yaml
---
phase: {phase-slug}
verified: {ISO datetime}
status: passed | failed
score: {N}/{M} must-haves verified
re_verification: false
---
```

### Body Sections
1. **Goal Achievement** — Observable truths from PLAN must_haves verified against code
2. **Required Artifacts** — Files expected vs files present
3. **Key Link Verification** — Integration wiring confirmed with line numbers
4. **Requirements Coverage** — Each requirement ID checked with evidence
5. **Anti-Patterns Found** — TODO/FIXME/placeholder scan
6. **Human Verification Required** — Items needing visual/runtime inspection
7. **Gaps Summary** — Overall assessment

### Evidence Style
- Specific file names and line numbers (approximate, can shift)
- Method signatures and code patterns found via grep
- Build verification output
