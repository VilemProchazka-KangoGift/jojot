  <name>Task 1: Fix file drop overlay over editor TextBox</name>
  <files>JoJot/MainWindow.xaml, JoJot/MainWindow.xaml.cs</files>
  <action>
Fix the file drop overlay so it appears when dragging files over the editor TextBox area.

**Root cause:** `AllowDrop="False"` on ContentEditor TextBox (MainWindow.xaml line 250) causes the TextBox's internal OLE drag handler to silently consume Win32 drag notifications with DROPEFFECT_NONE. This prevents WPF's tunneling routing from ever firing `PreviewDragEnter` on the Window when the mouse enters over the TextBox.

**Fix in MainWindow.xaml (line 250):**

Change:
```xml
AllowDrop="False"
```
To:
```xml
AllowDrop="True"
```

**Fix in MainWindow.xaml.cs -- add PreviewDragEnter/PreviewDragOver handlers on ContentEditor:**

The ContentEditor TextBox with AllowDrop="True" will now try to accept text drops (its default behavior). We need to suppress that while still allowing the tunneling events to propagate to the Window. Add handlers either in XAML or in the constructor:

In the constructor (after `InitializeComponent()`), add:
```csharp
ContentEditor.PreviewDragEnter += (s, e) =>
{
    // Allow file drops to propagate to Window handler; suppress text drops
    if (!e.Data.GetDataPresent(DataFormats.FileDrop))
    {
        e.Effects = DragDropEffects.None;
        e.Handled = true;
    }
};
ContentEditor.PreviewDragOver += (s, e) =>
{
    // Allow file drops to propagate to Window handler; suppress text drops
    if (!e.Data.GetDataPresent(DataFormats.FileDrop))
    {
        e.Effects = DragDropEffects.None;
        e.Handled = true;
    }
};
```

This ensures:
- File drags: tunneling events fire on Window, overlay appears, file drop works
- Text drags (e.g., selected text): suppressed at TextBox level, no text insertion
- The Window-level `OnFileDragEnter`/`OnFileDragOver` handlers already set `e.Handled = true` for file drops, preventing the TextBox from inserting file paths as text
  </action>

**Sub-issue 2: Unconditional _isDragOverlayActive guard blocks overlay refresh (line ~3568-3569)**

Replace the unconditional early return with context-aware logic. When the overlay is already active and a new move event comes in, we need to:
- If the new `toGuid` matches the original `_dragFromDesktopGuid` (moved back to original desktop): auto-dismiss the overlay.
- If the new `toGuid` is different from the current `_dragToDesktopGuid` (moved to a third desktop): update the overlay in-place.
- If the new `toGuid` is the same as `_dragToDesktopGuid` (same desktop, redundant event): return early (no-op).

Replace lines ~3568-3570:
```csharp
if (_isDragOverlayActive) return;
_isDragOverlayActive = true;
```

With:
```csharp
if (_isDragOverlayActive)
{
    // Moved back to original desktop -- auto-dismiss
    if (toGuid.Equals(_dragFromDesktopGuid, StringComparison.OrdinalIgnoreCase))
    {
        await DatabaseService.DeletePendingMoveAsync(_desktopGuid);
        _isMisplaced = false;
        if (Title.Contains(" (misplaced)"))
            Title = Title.Replace(" (misplaced)", "");
        await HideDragOverlayAsync();
        return;
    }
    // Same target desktop -- no-op
    if (toGuid.Equals(_dragToDesktopGuid, StringComparison.OrdinalIgnoreCase))
        return;
    // Different target desktop -- update overlay in-place (fall through)
    _dragToDesktopGuid = toGuid;
    _dragToDesktopName = toName;
    // Update pending_moves to new target
    await DatabaseService.DeletePendingMoveAsync(_desktopGuid);
    await DatabaseService.InsertPendingMoveAsync(_desktopGuid, _dragFromDesktopGuid!, toGuid);
    // Fall through to update UI below, but skip the initial setup parts
}
else
{
    _isDragOverlayActive = true;
    _dragFromDesktopGuid = fromGuid;
    _dragToDesktopGuid = toGuid;
    _dragToDesktopName = toName;
    await _autosaveService.FlushAsync();
    await DatabaseService.InsertPendingMoveAsync(_desktopGuid, fromGuid, toGuid);
}
```

Then the rest of the method (source label lookup, target name display, button visibility, overlay show) runs for both initial show and in-place update. For the in-place update case, the overlay is already visible so the fade-in animation will just re-confirm it at opacity 1.
