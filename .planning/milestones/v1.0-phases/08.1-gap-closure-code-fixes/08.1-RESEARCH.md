## Summary

This phase consists of two surgical code corrections in a single file (`MainWindow.xaml.cs`). All required infrastructure — `App.OpenWindowForOrphanAsync`, `AutosaveService.FlushAsync`, `CreateCardButton`, `RemoveOrphanGuid`, and `RefreshAfterOrphanAction` — already exists and is working. No new types, services, NuGet packages, or XAML changes are needed.

ORPH-03 requires inserting one button (Open) between the existing Adopt and Delete buttons in `CreateRecoveryCard`. The pattern is already established: call `CreateCardButton`, attach an `async` lambda click handler, call `App.OpenWindowForOrphanAsync(guid)`, then `RemoveOrphanGuid` + `RefreshAfterOrphanAction`. The only difference from Adopt/Delete is that OpenWindowForOrphanAsync does not modify the orphan's notes or session row — it opens the window directly, so the panel still refreshes like the other actions.

EDIT-03 requires changing `TabList_SelectionChanged` from `void` to `async void` and replacing the two-line `_autosaveService.Stop()` + `SaveCurrentTabContent()` sequence with a single `await _autosaveService.FlushAsync()` call. FlushAsync already stops the timer, saves content to the DB, updates `_lastWriteCompleted` (the frequency cap state), and pushes an undo snapshot — matching all behavior of the original code while correctly maintaining EDIT-03's frequency cap invariant.

**Primary recommendation:** Two atomic edits to `MainWindow.xaml.cs` at confirmed line numbers. No design work needed — both solutions are fully specified in CONTEXT.md.

## Architecture Patterns

### Pattern 1: Recovery Card Button (Established in Codebase)
**What:** Local function `CreateCardButton` creates a styled Button; an async lambda handles the click; cleanup calls `RemoveOrphanGuid` then `RefreshAfterOrphanAction`.
**When to use:** Every recovery panel action button.
**Example (current Adopt pattern):**
```csharp

var adoptBtn = CreateCardButton("Adopt");

{

    await DatabaseService.DeleteSessionAndNotesAsync(guid);
    RemoveOrphanGuid(guid);
    await RefreshAfterOrphanAction();
};
buttonPanel.Children.Add(adoptBtn);
```

**Open button pattern (applying the same template):**
```csharp

{
    await ((App)Application.Current).OpenWindowForOrphanAsync(guid);
    RemoveOrphanGuid(guid);
    await RefreshAfterOrphanAction();
};
buttonPanel.Children.Add(openBtn);
```

### Pattern 2: async void WPF Event Handler (Established in Codebase)
**What:** WPF event handlers returning `void` that need async work are declared `async void`. This is the project's established practice — see STATE.md decisions [06-01] (AutosaveService.OnTimerTick), [03-02] (HandleIpcCommand), [08-02] (confirmation callbacks).

### Pattern 3: FlushAsync Replaces Stop + SaveCurrentTabContent
**What:** `AutosaveService.FlushAsync()` is the correct flush path for tab switches. It calls `_timer.Stop()`, then awaits `DatabaseService.UpdateNoteContentAsync`, updates `_lastWriteCompleted`, pushes the undo snapshot, and invokes `_onSaveCompleted`. The old direct path (`SaveCurrentTabContent`) bypassed the service and left `_lastWriteCompleted` stale.

**Current code to replace (lines 420-433):**
```csharp
// Phase 6: Stop autosave timer and checkpoint timer before switching
_autosaveService.Stop();
_checkpointTimer.Stop();

// Phase 6: Save scroll offset for outgoing tab
if (_activeTab != null)
{
    var scrollViewer = GetScrollViewer(ContentEditor);
    if (scrollViewer != null)
        _activeTab.EditorScrollOffset = (int)scrollViewer.VerticalOffset;
}

// Save current tab content before switching
SaveCurrentTabContent();
```

**Target code (EDIT-03):**
```csharp
// Phase 6/8.1: Flush autosave service (stops timer, saves, updates frequency cap)
await _autosaveService.FlushAsync();
_checkpointTimer.Stop();

// Phase 6: Save scroll offset for outgoing tab
if (_activeTab != null)
{
    var scrollViewer = GetScrollViewer(ContentEditor);
    if (scrollViewer != null)
        _activeTab.EditorScrollOffset = (int)scrollViewer.VerticalOffset;
}
```

**Key**: `FlushAsync` calls `DatabaseService.UpdateNoteContentAsync` (which sets `_activeTab.Content` via the content provider), so the manual `SaveCurrentTabContent()` call becomes redundant and is removed. However, scroll offset save is NOT handled by FlushAsync — that line must be kept.

### Anti-Patterns to Avoid
- **Calling both FlushAsync AND SaveCurrentTabContent:** FlushAsync already saves via the content provider; calling both causes a double write and potentially incorrect state.
- **Removing _checkpointTimer.Stop():** The checkpoint timer is separate from the autosave timer; FlushAsync only stops `_autosaveService`'s internal timer, not `_checkpointTimer`.
- **Closing the recovery panel on Open click:** The CONTEXT.md decision is explicit: button should NOT close the recovery panel — it refreshes like the other actions.

## Common Pitfalls

### Pitfall 1: Scroll Offset Save Gets Lost
**What goes wrong:** Replacing the entire block including scroll offset save with only `await _autosaveService.FlushAsync()` removes the scroll offset preservation for the outgoing tab.
**Why it happens:** FlushAsync handles content save only, not UI state like scroll offset.
**How to avoid:** Keep the `if (_activeTab != null) { ... scrollViewer ... }` block after the FlushAsync call; only remove `_autosaveService.Stop()` and `SaveCurrentTabContent()`.
**Warning signs:** EDIT-05 (scroll offset restore on tab restore) will fail — scrolling back to a tab lands at position 0.

### Pitfall 2: Double-Write on Tab Switch
**What goes wrong:** If `SaveCurrentTabContent()` is left in alongside `FlushAsync()`, two DB writes fire per tab switch — first from FlushAsync (async, correct), then from SaveCurrentTabContent (fire-and-forget, sets stale `_lastWriteCompleted` shadow state via direct DB call).
**Why it happens:** SaveCurrentTabContent does its own `_ = DatabaseService.UpdateNoteContentAsync(...)` call without going through AutosaveService.
**How to avoid:** Remove `SaveCurrentTabContent()` entirely from `TabList_SelectionChanged` — FlushAsync covers it.
**Warning signs:** Tab switch latency slightly higher; `_lastWriteCompleted` state incorrect.

### Pitfall 3: App Cast for OpenWindowForOrphanAsync
**What goes wrong:** Calling `OpenWindowForOrphanAsync` without the correct cast — the method is on `App`, not `Application`.
**Why it happens:** `Application.Current` returns `Application` base type; the method is on the derived `App` class.
**How to avoid:** Cast: `((App)Application.Current).OpenWindowForOrphanAsync(guid)`.
**Warning signs:** Compilation error — `Application` does not contain a definition for `OpenWindowForOrphanAsync`.

### Pitfall 4: Open Button Closes Panel Unintentionally
**What goes wrong:** Following the Adopt/Delete cleanup too literally and calling `HideRecoveryPanel()` or similar.
**Why it happens:** Adopt and Delete reduce the orphan count, which can trigger `HideRecoveryPanel()` inside `RefreshAfterOrphanAction()`. Open does the same (it calls `RemoveOrphanGuid` + `RefreshAfterOrphanAction`) — so the panel hides if this was the last orphan. This is correct. The constraint is: don't add an explicit `HideRecoveryPanel()` call in addition to the shared refresh path.
**How to avoid:** Follow the Adopt pattern exactly — `RemoveOrphanGuid` then `RefreshAfterOrphanAction`. Let `RefreshAfterOrphanAction` decide whether to close the panel based on remaining orphan count.

### Pitfall 5: Stale _activeTab During FlushAsync
**What goes wrong:** In `TabList_SelectionChanged`, the content provider lambda captures `_activeTab` by reference at the time of the call. `FlushAsync` calls `_contentProvider()` which in turn reads `_activeTab`. If `_activeTab` were changed before the flush, the wrong content would be saved.
**Why it happens:** N/A — this pitfall does NOT apply here. `_activeTab` is only updated on line 440 (AFTER the flush block at lines 420-433). The ordering is safe.
**How to avoid:** No action needed — just confirming safe execution order.

## Open Questions

1. **Does FlushAsync interact correctly with _suppressTextChanged?**
   - What we know: FlushAsync calls `_contentProvider()`, which returns `(_activeTab.Id, ContentEditor.Text)`. It does not set `ContentEditor.Text`, so `_suppressTextChanged` is irrelevant.
   - What's unclear: nothing — confirmed safe.
   - Recommendation: No action needed.

2. **Can OpenWindowForOrphanAsync be called safely when the orphan's GUID is already in _windows?**
   - What we know: App.xaml.cs:288-291 checks `_windows.TryGetValue(orphanGuid, ...)` and if found, activates the existing window instead of creating a new one. This is graceful.
   - What's unclear: nothing — confirmed safe.
   - Recommendation: No action needed.

3. **Should Open button use await or fire-and-forget?**
   - What we know: The click handler lambda is already `async (s, e)` (matching Adopt/Delete). `OpenWindowForOrphanAsync` is `Task`-returning. Use `await` to ensure the window is created before calling `RemoveOrphanGuid`.
   - Recommendation: Use `await` (matching the other buttons' pattern).
