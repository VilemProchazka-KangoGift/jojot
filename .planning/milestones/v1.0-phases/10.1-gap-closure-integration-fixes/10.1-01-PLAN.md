  <action>
  Two fixes in MainWindow.xaml.cs:

  **Fix 1 — OnClosing (line ~1650):**
  Add `CommitPendingDeletionAsync().GetAwaiter().GetResult()` to the OnClosing method. Place it AFTER the autosave stop and content flush block, but BEFORE the geometry capture. This ensures pending tab deletions are committed to DB before the window closes.

  Insert after the content flush block (after line ~1669, the closing brace of `if (_activeTab != null)`) and before the geometry capture (line ~1672):
  ```csharp
  // Phase 10.1: Commit pending deletions before close (audit finding #1)
  CommitPendingDeletionAsync().GetAwaiter().GetResult();
  ```

  This follows the established pattern — `UpdateNoteContentAsync(...).GetAwaiter().GetResult()` is already used on line ~1667-1668 for synchronous content flush. No deadlock risk because CommitPendingDeletionAsync only does DB operations (no UI thread marshaling). The null guard inside CommitPendingDeletionAsync (`if (_pendingDeletion == null) return;`) makes this safe even when no deletion is pending.

  **Fix 3 — FlushAndClose (line ~1627):**
  Change the fire-and-forget `_ = CommitPendingDeletionAsync();` (line ~1640) to synchronous execution:
  ```csharp
  // Phase 10.1: Await pending deletion before close (audit finding #3)
  CommitPendingDeletionAsync().GetAwaiter().GetResult();
  ```

  This ensures deletion completes before `Close()` is called, which prevents the race condition where `ExitApplication` calls `Environment.Exit(0)` before the deletion DB writes finish. Update the comment on line 1638-1639 to remove the "fire-and-forget" language.
