  <name>Task 1: Add _isTransferringCapture guard to prevent LostMouseCapture from aborting drag on capture transfer</name>
  <files>JoJot/MainWindow.xaml.cs</files>
  <action>
This task fixes UAT Tests 2 and 3 by addressing the primary root cause diagnosed in `.planning/debug/tab-drag-opacity-issues.md`.

**Root cause:** When drag starts, `Mouse.Capture(TabList, CaptureMode.SubTree)` transfers capture from the individual ListBoxItem to TabList. This fires `LostMouseCapture` on TabList (it bubbles up from the ListBoxItem). The handler sees `_isDragging=true` and calls `CompleteDrag()`, aborting the drag within milliseconds.

**Fix — Three changes to JoJot/MainWindow.xaml.cs:**

1. **Add a new class-level field** next to the existing `_isCompletingDrag` field (around line 32):
   ```csharp
   private bool _isTransferringCapture; // R3-REORDER-01: Guard for Mouse.Capture transfer during drag start
   ```

2. **Guard the LostMouseCapture handler** (around line 146-157). Add `_isTransferringCapture` check alongside the existing `_isCompletingDrag` check:
   ```csharp
   TabList.LostMouseCapture += (s, e) =>
   {
       // R2-DND-01: Prevent re-entrant cleanup (Mouse.Capture(null) in CompleteDrag can re-fire)
       // R3-REORDER-01: Skip when capture is being intentionally transferred to TabList
       if (_isCompletingDrag || _isTransferringCapture) return;

       if (_isDragging)
       {
           RemoveDropIndicator();
           if (_dragItem != null) _dragItem.Opacity = 1.0;
           CompleteDrag();
       }
   };
   ```

3. **Wrap Mouse.Capture in drag start** with the guard flag (around lines 1408-1410 in TabItem_PreviewMouseMove). Change:
   ```csharp
   Mouse.Capture(TabList, CaptureMode.SubTree);
   ```
   To:
   ```csharp
   _isTransferringCapture = true;
   try
   {
       Mouse.Capture(TabList, CaptureMode.SubTree);
   }
   finally
   {
       _isTransferringCapture = false;
   }
   ```

**Do NOT change** the CompleteDrag method, the fade-in animation code (lines 1573-1592), or the hover handlers — those are already correct. The only problem was that the drag was being aborted before those code paths could run.

**Why this approach:** The guard flag is minimal and surgical. It only affects the specific moment when capture transfers during drag start. All other LostMouseCapture scenarios (genuine loss, window deactivation, etc.) continue to work as before. This mirrors the existing `_isCompletingDrag` pattern already used for the Mouse.Capture(null) case in CompleteDrag.
