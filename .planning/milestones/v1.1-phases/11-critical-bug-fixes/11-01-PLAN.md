<task type="auto">
  <name>Task 1: Fix BUG-01 and BUG-02 — Add re-entry guard to stop event cascade crashes</name>
  <files>JoJot/MainWindow.xaml.cs</files>
  <action>
Root cause: `RebuildTabList()` re-hooks `SelectionChanged` then calls `SelectTabByNote(_activeTab)`, which sets `TabList.SelectedItem` and fires `TabList_SelectionChanged`. This async void handler can cause re-entrant calls when selection changes trigger further selection changes. Additionally, `TogglePinAsync` calls `SelectTabByNote(tab)` AFTER `RebuildTabList()` returns — a duplicate that fires the handler a second time unnecessarily.

**Change 1 — Add guard field** in the field declarations section (after `_isDragging`):
```csharp
private bool _isRebuildingTabList;
```

**Change 2 — Guard `TabList_SelectionChanged`** — add an early return at the very top of the handler body (before any other logic):
```csharp
private async void TabList_SelectionChanged(object sender, SelectionChangedEventArgs e)
{
    if (_isRebuildingTabList) return;   // <-- ADD THIS LINE
    // Remove accent from deselected items
    foreach (var removed in e.RemovedItems)
    ...
```

**Change 3 — Wrap `RebuildTabList()` body with the guard**. Set `_isRebuildingTabList = true` immediately after the `TabList.SelectionChanged -= TabList_SelectionChanged` unhook line, and set it back to `false` immediately after the `TabList.SelectionChanged += TabList_SelectionChanged` rehook line (BEFORE the `SelectTabByNote` call):
```csharp
private void RebuildTabList()
{
    TabList.SelectionChanged -= TabList_SelectionChanged;
    _isRebuildingTabList = true;       // <-- ADD
    TabList.Items.Clear();
    // ... existing loop logic unchanged ...
    TabList.SelectionChanged += TabList_SelectionChanged;
    _isRebuildingTabList = false;      // <-- ADD (before SelectTabByNote)

    // Re-select active tab if still visible
    if (_activeTab != null)
        SelectTabByNote(_activeTab);
}
```

**Change 4 — Remove duplicate `SelectTabByNote` from `TogglePinAsync`**. `RebuildTabList()` already calls `SelectTabByNote(_activeTab)`. The external call to `SelectTabByNote(tab)` on the line after `RebuildTabList()` is redundant (`tab` IS `_activeTab` in this context). Delete this line:
```
SelectTabByNote(tab);   // DELETE THIS LINE from TogglePinAsync
```
The `UpdateToolbarState()` call that follows must be kept.

After these changes, the flow is: `TogglePinAsync` calls `RebuildTabList()` → guard set → items rebuilt → guard cleared → `SelectTabByNote` fires `SelectionChanged` → handler fires once with `_isRebuildingTabList == false` → no re-entry possible. `TogglePinAsync` no longer calls `SelectTabByNote` again.

For delete (`DeleteTabAsync` and `DeleteMultipleAsync`): `RebuildTabList()` is called, then `ApplyFocusCascadeAsync` calls `SelectTabByNote`. Since the guard is cleared before `SelectTabByNote` in `RebuildTabList`, and `ApplyFocusCascadeAsync` calls `SelectTabByNote` AFTER `RebuildTabList` returns, this fires one clean `SelectionChanged`. No change needed to delete paths beyond the guard.

<task type="auto">
  <name>Task 2: Fix BUG-03 — Extend SelectionChanged unhook to cover SelectedItem assignment in UpdateTabItemDisplay</name>
  <files>JoJot/MainWindow.xaml.cs</files>
  <action>
Root cause: In `UpdateTabItemDisplay`, the `TabList.SelectionChanged` event is unhooked around `TabList.Items[index] = newItem` but NOT around `TabList.SelectedItem = newItem`. Setting `SelectedItem` fires `TabList_SelectionChanged`, which runs async code including `await _autosaveService.FlushAsync()`. When `UpdateTabItemDisplay` is called from within `CommitRename` (which is called from `RenameBox_LostFocus`), this async event handler firing on the UI thread while focus is mid-change causes a freeze loop.

**Change — Extend the unhook bracket** in `UpdateTabItemDisplay` to include the `TabList.SelectedItem = newItem` assignment:

Current code:
```csharp
TabList.SelectionChanged -= TabList_SelectionChanged;
TabList.Items[index] = newItem;
TabList.SelectionChanged += TabList_SelectionChanged;

if (wasSelected)
{
    TabList.SelectedItem = newItem;     // fires SelectionChanged — NOT guarded
    ApplyActiveHighlight(newItem);
}
```

Replace with:
```csharp
TabList.SelectionChanged -= TabList_SelectionChanged;
TabList.Items[index] = newItem;
if (wasSelected)
{
    TabList.SelectedItem = newItem;     // now guarded — no SelectionChanged fired
}
TabList.SelectionChanged += TabList_SelectionChanged;

if (wasSelected)
{
    ApplyActiveHighlight(newItem);
}
```

This prevents `TabList_SelectionChanged` from firing during the item replacement. The `ApplyActiveHighlight` call (which only sets visual brush properties) remains outside and executes after the event is re-hooked — it does not need to be guarded and must not be removed.

No changes to `CommitRename`, `BeginRename`, `CancelRename`, or `RenameBox_LostFocus` — the fix is localized to `UpdateTabItemDisplay`.
